# Живая Доказательная База

Этот файл фиксирует не синтетические toy-примеры, а дистиллированные archetypes, извлечённые из большой рабочей 1С-кодовой базы.
Исходный репозиторий, его путь и project-specific identifiers здесь намеренно не раскрываются.

## Как читать этот файл

- Каждый archetype основан на реально встречающемся сочетании `ConfigDump`, форм, команд, модулей и табличных частей.
- Названия объектов ниже обезличены; они описывают форму задачи, а не конкретную конфигурацию.
- Цель файла не пересказать чужую кодовую базу, а доказать, что `onec-context-toolkit` покрывает реальные developer questions, а не только демонстрационные кейсы.

## Наблюдаемый живой baseline

- Источник оказался `single-target` конфигурацией, а не toy-tree с парой файлов.
- В дереве есть плотный слой metadata-объектов: задачи, справочники, отчёты, общие команды, версии форм и клиентские command modules.
- В metadata snapshot собралось около `219k` строк индекса при объёме входа около `113 MB`.
- В формах реально встречаются большие command-heavy интерфейсы с действиями вроде выбора периода, формирования, выгрузки, отправки, сравнения и показа настроек.
- В metadata встречаются табличные части и generated row types для отчётов и справочников.
- В коде встречаются клиентские обработчики команд, которые открывают формы и маршрутизируют пользователя в специализированные UI-сценарии.

## Distilled Archetypes

### 1. Объект задачи с исполнителем и персональными действиями

- Реальный shape:
  - объект вида `Task`
  - набор реквизитов автора, предмета, сроков и описания
  - отдельные командные действия для персонального списка задач
  - клиентский command module, который открывает специализированную форму
- Тип developer question:
  - где лежат реквизиты карточки задачи
  - как открыть персональную форму по команде
  - какая логика реально выполняется из обработчика команды
- Минимальный route:
  - `metadata` для реквизитов и структуры
  - `code` для обработчика команды и открытия формы
- Что toolkit должен доказать:
  - `confirmed facts`: наличие task-объекта, реквизитов и команды
  - `code-derived inferences`: какой UI-сценарий открывает обработчик
  - `unresolved assumptions`: что происходит уже внутри открытой формы, если не делался дальнейший code/full drill-down

### 2. Аналитический отчёт с плотной формой и табличными частями

- Реальный shape:
  - объект вида `Report`
  - реквизиты периода, организации и режимов отображения
  - несколько табличных частей
  - большая форма отчёта с десятками `Form.Command.*`
- Тип developer question:
  - какие настройки и действия доступны пользователю на форме
  - где в metadata лежат группировки и дополнительные поля
  - что нужно проверить, чтобы подтвердить точное расположение команд в XML формы
- Минимальный route:
  - `metadata` для реквизитов и табличных частей
  - `full` для lossless подтверждения exact form XML
- Что toolkit должен доказать:
  - `confirmed facts`: наличие отчёта, табличных частей и command-heavy формы
  - `code-derived inferences`: здесь не обязательны до тех пор, пока вопрос не касается поведения команд
  - `unresolved assumptions`: какие команды только объявлены в форме, но ещё не прослежены до обработчиков

### 3. Регламентированная форма с жизненным циклом "сохранить -> проверить -> выгрузить -> отправить"

- Реальный shape:
  - объект отчётного типа с versioned forms
  - повторяющиеся UI-действия по сохранению, проверке, выгрузке, сравнению и отправке
  - несколько поколений форм под разные периоды или релизы
- Тип developer question:
  - как устроен жизненный цикл формы
  - какие действия доступны пользователю до отправки
  - где подтверждать exact UI-состав для конкретной версии формы
- Минимальный route:
  - `metadata` для формы и связанных реквизитов
  - `full` для version-exact XML формы
- Что toolkit должен доказать:
  - `confirmed facts`: набор действий и наличие versioned forms
  - `code-derived inferences`: выводы о последовательности пользовательского сценария допустимы только после code/full follow-up
  - `unresolved assumptions`: что именно делает каждая команда серверно, если code layer ещё не использовался

### 4. Справочник с табличными секциями и файловыми/служебными действиями

- Реальный shape:
  - объект вида `Catalog`
  - несколько табличных частей с row types
  - форма списка или карточка с предметными командами
- Тип developer question:
  - где в metadata лежит табличная секция
  - какие поля относятся к строке табличной части
  - когда нужен `full`, чтобы подтвердить file-level представление объекта
- Минимальный route:
  - `metadata` для object shape
  - `full` для точного XML-подтверждения
- Что toolkit должен доказать:
  - `confirmed facts`: наличие справочника, секций и row-level структуры
  - `code-derived inferences`: не обязательны, если вопрос только про структуру
  - `unresolved assumptions`: как UI-команды влияют на данные без code layer

### 5. Плотный слой общих команд как отдельная proof surface

- Реальный shape:
  - десятки и сотни `CommonCommands/*`
  - команды охватывают навигацию, настройки, интеграции, сервисные и предметные сценарии
- Тип developer question:
  - насколько toolkit применим не только к объектам metadata, но и к общей командной поверхности конфигурации
  - как доказывать, что команда относится к форме, отчёту или отдельному UI-flow
- Минимальный route:
  - `metadata` для обнаружения самой командной поверхности
  - `code` для прослеживания обработчиков
- Что toolkit должен доказать:
  - `confirmed facts`: что в живой базе есть широкая command surface, а не только isolated objects
  - `code-derived inferences`: какие команды реально маршрутизируют пользователя в формы или процессы
  - `unresolved assumptions`: что делает команда после входа в downstream object без дополнительного follow-up

## Почему это подтверждает жизнеспособность skill

- Proof surface теперь опирается не только на канонические примеры из README, но и на patterns, которые реально встречаются в большой production-like кодовой базе.
- Distilled archetypes покрывают разные уровни сложности: чистая metadata-структура, command-heavy формы, code-backed UI-navigation и lossless XML confirmation.
- Из этих archetypes видно, что `platform / metadata / code / full` модель не искусственная: каждый слой закрывает отдельный класс вопросов.
- Одновременно становится видно и operational ограничение: на большой живой базе нужно фиксировать не только happy path, но и стоимость/стадии сборки каждого слоя.
